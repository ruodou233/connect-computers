# 多机运维方法论（长期实践沉淀）

> 以下是作者本机多机协作的运维方法论，不是照抄即用的教程，具体工具选型和配置请你的 Agent 按你自己的设备情况判断。
>
> 这份内容是本 skill 首次配置方案之外的补充：首次配置解决"怎么把几台电脑连起来"，这里记录的是长期多机运维中沉淀下来的分层思路、工具选择原则、验收清单和安全边界，来自作者本人日常使用中的实践总结。真实设备清单、SSH alias、IP、账号信息不在此列，仅保留可以脱敏复用的方法论部分。

## 分层架构

把多台电脑连成一套可维护的远程工作网络，建议拆成五层分别设计和验收，而不是混在一起：

1. **网络层**：VPN（如 Tailscale）/ 内网双入口。内网优先走局域网地址；离家或跨网段走 VPN。
2. **命令层**：SSH 是主入口。所有可自动化的安装、更新、配置、检查优先走 SSH。
3. **屏幕层**：远程桌面只在需要 GUI、网页登录、首次授权、游戏启动器、商店应用时使用。
4. **Agent 层**：每台电脑独立安装 Agent CLI 或桌面端；只同步非密钥配置。
5. **供电层**：优先"显示器快速关闭，主机不自动睡眠"。Wake-on-LAN 只能作为备用能力，不能替代在线主入口。

## 工具选择原则

### 命令优先于屏幕

能用命令完成的事不要打开远程桌面：装包、改配置、查版本、关服务、同步文件都优先 SSH。屏幕层只用于必须要 GUI 的场景：首次登录、系统权限弹窗、应用商店/游戏启动器、图形设置、远程确认画面。

手机控制电脑主要依赖同一套 VPN + 远控客户端；手机适合应急点击和查看状态，不适合长时间复杂维护。远控工具不要作为唯一入口；至少保留 SSH + VPN 两条命令入口。

### 电源策略

核心目标是"屏幕不一直亮，但 Agent 随时能连上干活"：

- 显示器：较短时间后关闭。
- 主机睡眠/休眠：需要随时访问的机器设为不自动睡眠。
- 外设灯光：尽量只关闭灯光，不禁用键盘/鼠标本身。
- Wake-on-LAN：可配置，但必须实测。它依赖网卡、主板、路由器、同网段中继和系统电源状态，不保证跨公网稳定唤醒。

### 无头屏幕方案的选择顺序

> 以下信息核实于 2026-07（用户当前会话日期），涉及具体系统版本要求和各工具能力边界，请自行核实当前情况后再决定。

无头 Mac（不接显示器）想在另一台设备上直接看到并控制桌面时，建议按下面的顺序叠加，而不是一上来就上最复杂的方案：

1. **首选：系统内置的远程屏幕高性能模式**（如 macOS Screen Sharing High Performance）
   - 不是开源方案，但通常是同生态内无头控制最少依赖、最稳定的起点。
   - 典型适用条件：两端都是同代芯片，且系统版本较新（例如作者核实时的条件是 Apple silicon + macOS Sonoma 14 或更新版本，支持虚拟显示器）。
   - 风险：网络差时需要降级到标准模式；磁盘加密、重启后未解锁、系统睡眠都会影响可达性。

2. **传输层：VPN + 系统内置远程屏幕**
   - VPN（如基于 WireGuard 的 Tailscale）承载远程屏幕/远控/SSH 等任一入口，而不是和内置方案二选一。
   - 优点：不用公网端口映射，远程屏幕类服务不裸露到公网。
   - 风险：画质和延迟取决于网络；如果走中继节点，体验会下降。

3. **分辨率/无头补强：虚拟显示器工具**（如 BetterDisplay）
   - 源码公开，社区常用于 headless 场景的虚拟显示和分辨率修正。
   - 适用场景：内置高性能模式不可用，或标准远控能连但分辨率不合适、没有物理显示器时桌面尺寸不理想。
   - 风险：不同系统版本下虚拟屏、鼠标残影、权限入口可能变化；需要首次 GUI 配置。

4. **开源跨平台备选：RustDesk 类方案（结合 VPN 内网地址）**
   - 开源远控，支持自建 relay/server，也可直接走 VPN 内网地址。
   - 适用场景：需要跨平台控制多种系统，或内置方案不够顺手。
   - 风险：macOS 需要屏幕录制、辅助功能、输入监控等权限；无头机器可能仍遇到"没有可捕获显示"的问题，通常要配虚拟显示器工具或 HDMI dummy 假负载。

5. **低延迟/高画质备选：Sunshine + Moonlight 类串流方案**
   - 开源串流方案，社区评价集中在低延迟、高码率、游戏/视频体验。
   - 风险：作为无头机器的远程管理入口，配置复杂度通常高于纯管理场景，权限、音频、虚拟显示器经常需要额外配置；不作为纯管理节点的首选。

6. **只共享键鼠的方案**（如 Input Leap / Barrier）
   - 开源键鼠共享工具，不传画面。适用于目标机器已经接了显示器、只想共享键鼠的场景；不能满足"用这边屏幕看那边桌面"的核心需求。

7. **商业备选**（如 NoMachine / AnyDesk 一类工具）
   - 内置方案不稳定、手机体验优先、需要更省心的 NAT 穿透时可以作为救急或体验优化，但不建议作为主架构的唯一入口。

选择顺序建议：先实测系统内置方案 → 需要离家访问就把它放到 VPN 上而不开放公网端口 → 分辨率不合适再叠加虚拟显示器工具 → 需要手机/跨平台方便控制再补开源远控并限制走 VPN → 需要低延迟高画质再单独试串流方案，不和基础远程管理绑定 → 如果"开源优先"是硬约束，走开源远控或串流方案 + 虚拟显示器工具，代价是权限、音频、虚拟显示器和无头捕获更复杂。

无头场景的通用注意事项：

- 关闭自动睡眠，只让显示器关闭；否则远端屏幕方案会跟着失联。
- 保留独立 SSH 作为带外入口；屏幕方案不能成为唯一命脉。
- 磁盘加密（如 FileVault）可能导致重启后远程登录前不可用；无人值守重启前应评估好这个取舍。
- 首次开启远程屏幕、远控、虚拟显示器工具时通常需要物理屏幕或已有远控完成系统权限授权。
- HDMI dummy 是最后的硬件兜底：当虚拟屏/系统屏幕共享不稳定时，用它模拟显示器存在。

## 验收清单

新增一台设备接入多机网络时，建议按这个顺序逐项验收，而不是配完就默认可用：

1. 能从主控设备 SSH 到新机器。
2. 能通过 VPN 地址或 VPN DNS 到新机器。
3. 能查看远程屏幕。
4. 能安装并验证 Agent CLI（如 Codex / Claude Code）。
5. 能同步非密钥配置。
6. 能确认登录状态。
7. 能确认电源策略不会让机器自动失联。
8. 能确认新设备可以访问到你自己维护的共享配置/操作手册（同步方案由你自己决定，比如共享 git 仓库、云盘等）。

## 安全边界

### Agent 配置同步边界

多机之间同步 Agent 配置时，明确区分可同步和不可同步的内容：

**可以同步：**

- 模型选择
- 推理强度 / effort 档位
- 审批策略（approval policy）
- 沙盒模式（sandbox mode）
- 服务层级（service tier）
- 插件启用项
- 项目信任（trusted）配置
- 自建 skills / references / 操作手册

**不能同步：**

- 各平台（ChatGPT / OpenAI / Claude / Google 等）的登录态
- token、session、Cookie、OAuth code
- 浏览器个人资料目录
- 一次性验证码或恢复码
- 明文密码

账号相同不等于会话可复制。新机器需要用官方登录流程重新授权；Agent 只能验证登录状态，不能绕过平台安全机制。

### 临时远控服务的清理

临时开启的 VNC/RDP/弱口令服务用完必须停用或卸载，不留常驻的低安全性入口。

### 变更安全顺序（排障时避免制造新故障）

远控/网络类问题排查时，建议在改动前、中、后各做一段，防止排障本身引入新故障：

1. **先验证基础连通，再并列验证各服务**：先确认组网层（VPN/内网、ping/直连）可达；组网通了之后 SSH、远程屏幕、文件共享是并列的上层服务，分别独立验证，只修确认失败的那个服务；组网层不通时不要先动服务层配置。
2. **防火墙只开具体程序/端口**，不整体关闭防火墙、不加宽泛网段放行；操作完成后回验规则确实生效且未多开。
3. **代理类故障拆两类分别处理**：①端口监听冲突——先查端口占用定位互抢的两个进程，退出/关闭抢占来源进程本身，来源还在只改端口号会再次相撞；②系统代理配置被反复覆盖——找到会回写系统代理设置的客户端，关其回写功能而不是只改系统层设置，否则改完即被覆盖。
4. **改系统代理/网络配置前记录原值**（注册表键/配置项/开关状态），改后验证目标功能，失败即按记录回滚；一次只改一处，改多处无法定位是哪处生效。

### 隐私与分享的一般原则

这类多机运维文档天然会积累大量个人环境细节（设备别名、真实设备名、内网/VPN IP、账号邮箱、能定位个人网络拓扑的信息）。如果你要把自己的操作手册分享给别人，建议只保留分层架构、工具选择原则、验收清单和安全边界这几类可以脱敏复用的内容，个人环境细节留在本地、不进入可分享文档。
